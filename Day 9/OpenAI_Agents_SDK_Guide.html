Overview

Introduction &
High-Level Architecture

The OpenAI Agents SDK is OpenAI's official framework for building AI systems that go beyond single-prompt completions — purpose-built for agents that reason in steps, call tools, hand work off to other agents, hold persistent memory, and run inside production pipelines.

What You Can Build

  • AI agents that reason, decide, and act autonomously
  • Multi-agent systems with specialized roles
  • Tool-using assistants that call APIs, databases, or functions
  • Autonomous workflows with branching logic
  • Stateful conversations with persistent memory
  • Agent-to-agent handoffs and delegation

Design Principles

  • Lightweight — no heavy framework overhead
  • Modular — compose from small, swappable pieces
  • Production-ready — async-first, streaming, guardrails
  • Easy to orchestrate — scale from 1 to N agents without rewriting foundations
💡 Mental model: think of the SDK as an operating system for AI workers. The Runner is the scheduler, Agents are processes, Tools are system calls, and Memory is shared state.

02
Core Component

Agent
The Central AI Worker

An Agent bundles together everything needed to reason and act: a name, instructions, a model, a set of tools, optional memory, optional handoff targets, and optional guardrails.

An Agent is best understood as an AI employee with defined skills and responsibilities.
Python
from agents import Agent research_agent = Agent( name="Research Agent", instructions="You are an expert at researching AI trends. Be concise and cite sources.", model="gpt-4.1-mini", tools=[search_web, summarize], output_type=ResearchReport, # optional Pydantic schema guardrails=[no_pii_guardrail], # optional safety checks )

Constructor Parameters & Properties

name required
str
Identity string for this agent. Used in logs, handoff references, debugging traces, and multi-agent routing. Should be unique and descriptive within your system.
instructions required
str
System prompt that defines the agent's behavior, persona, constraints, and focus area. This is the most important configuration — write it with precision. Supports Markdown formatting.
Best practice: be specific, give examples, define limits
model required
str
Underlying LLM the agent runs on. Different agents in the same system can use different models — e.g., a fast cheap model for routing and a powerful model for analysis.
Examples: "gpt-4.1-mini", "gpt-4o", "gpt-4-turbo"
tools optional
list[Tool]
List of callable tools this agent can invoke. Each tool is a Python function decorated with @function_tool. The agent decides when and which tool to call based on context.
Default: [] (no tools — pure LLM)
handoffs optional
list[Agent]
Other agents this agent can delegate work to. When this agent encounters a task better suited to another specialist, it performs a handoff. The runner manages the delegation and returns results.
Default: [] (no delegation)
memory optional
Memory
Memory provider for short-term or persistent context. Enables the agent to recall previous turns, user preferences, and facts across invocations. Can be scoped per session or globally.
Default: None (stateless)
output_type optional
Type[BaseModel]
Pydantic model class that enforces structured, typed output instead of free text. Enables agents to return JSON objects that integrate directly with downstream systems and APIs.
Default: None (free-text response)
guardrails optional
list[Guardrail]
Validation & safety checks applied to agent outputs before they pass downstream. Can validate structure, block unsafe content, enforce policy, or restrict tool access in given contexts.
Default: [] (no validation)

03
Execution Engine

Runner
The Orchestrator

The Runner is the most important component in the SDK. It turns a configured Agent into a running system — without it, agents are inert definitions. It drives tool calling, memory updates, handoffs, streaming, and context propagation.

If the Agent is the worker, the Runner is the manager. Without it, nothing actually runs.
Python
from agents import Runner # Synchronous (simple scripts) result = Runner.run_sync(agent, "Latest AI trends in 2025") print(result.final_output) # Asynchronous (APIs, production systems) result = await Runner.run(agent, "Latest AI trends in 2025") # Streaming (real-time UX) async for chunk in Runner.run_streamed(agent, "Latest AI trends"): print(chunk, end="", flush=True)

Key Methods & Parameters

Method
Returns
Description
run_sync(agent, input)
RunResult
Synchronous blocking call. Simplest way to run an agent. Blocks until the agent completes. Ideal for scripts and CLIs. Parameters: agent: Agent, input: str | list, context: dict = None
run(agent, input)
Coroutine[RunResult]
Async execution. Non-blocking coroutine for production APIs and parallel workflows. Must be awaited. Plays well with FastAPI, asyncio. Parameters: agent: Agent, input: str | list, context: dict = None
stream(agent, input)
AsyncGenerator[str]
Token-level streaming. Yields tokens as they are generated. Essential for chat UIs, live dashboards, perceived-latency reduction. Parameters: agent: Agent, input: str, stream_options: dict = None
run_streamed(agent, input)
AsyncGenerator[RunEvent]
Full workflow streaming. Streams the entire agent workflow incrementally — tool calls, intermediate steps, and final output. Exposes RunEvents for fine-grained control. Parameters: agent: Agent, input: str, context: dict = None

Execution Flow

📥
INPUT RECEIVED
Runner.run(agent, input) is called — begins the execution loop
🧠
AGENT REASONS
Agent LLM processes instructions + input + memory context and decides next action
⚙️
TOOL CALL (if needed)
Runner detects tool_call in response, executes the function, injects result back into reasoning
🔄
CONTINUE LOOP
Agent continues reasoning with new information. Loop repeats until no more tool calls needed
📤
FINAL OUTPUT
result.final_output returned — either free text or typed Pydantic object

Capabilities

Tools
Giving Agents Real Capabilities

Tools are functions that agents can call. They are the difference between an agent that generates text and an agent that does things. A pure LLM can write a description of the weather; an agent with a weather tool can fetch actual conditions.

Python — Defining & Attaching Tools
from agents import function_tool, Agent @function_tool def get_weather(city: str) -> dict: """Fetch current weather for a city. Returns temperature and condition.""" # Your API call here return {"city": city, "condition": "Sunny", "temp_c": 28} @function_tool def search_web(query: str, max_results: int = 5) -> list[str]: """Search the web for current information.""" return [...] weather_agent = Agent( name="Weather Agent", tools=[get_weather, search_web] )

@function_tool Decorator — Properties

Function signature required
callable
Type-annotated Python function. Parameter names and type hints are automatically converted to the tool's JSON schema, which is passed to the LLM so it knows how to call the tool correctly.
Docstring optional
str
Tool description shown to the LLM. Helps the model understand when to use the tool and what it does. Better docstrings → better tool selection decisions.
Return type optional
any JSON-serializable
Output injected back into the agent's reasoning context. Must be JSON-serializable: str, int, list, dict, or Pydantic model. Non-serializable types will raise errors at runtime.
🌐
Web Search
Query search engines and retrieve current, real-time information beyond the model's training cutoff
🗄️
Database Queries
Run SQL or NoSQL queries against your data stores, retrieve records, and return structured results
💻
Code Execution
Execute Python scripts, run calculations, process data, or trigger any subprocess in your environment
🔌
External APIs
Call any REST or GraphQL endpoint — payment processors, CRMs, communication platforms, IoT devices
📁
File Operations
Read from and write to the filesystem, process documents, generate reports, or manage data files
⚡
Business Workflows
Trigger approvals, send notifications, update records, or orchestrate multi-step business processes
05
Multi-Agent

Handoffs
Multi-Agent Delegation

Handoffs let you compose teams of specialists that delegate to each other, rather than building one monolithic agent that does everything. Each agent stays focused on what it does best.

Rather than one giant agent that does everything, compose teams of specialists that delegate to each other. Systems scale by adding agents, not by bloating one prompt.
Python — Multi-Agent System
research_agent = Agent( name="Researcher", instructions="Find facts, data, and sources on any given topic.", tools=[search_web, fetch_url] ) writer_agent = Agent( name="Writer", instructions="Transform research findings into polished prose.", output_type=Article ) manager = Agent( name="Manager", instructions="Coordinate research and writing. Route tasks to the right specialist.", handoffs=[research_agent, writer_agent] ) result = Runner.run_sync(manager, "Write a 500-word article on quantum computing")

Handoff Configuration Properties

handoffs on Agent
list[Agent]
List of target agents this agent can delegate to. The delegating agent decides which specialist to call based on the task. The runner manages the invocation and returns results to the originating agent.
Agents must be instantiated before being referenced in handoffs
Delegation decision
automatic (LLM)
The model decides when and which agent to hand off to based on agent names, instructions, and the current task context. Write clear agent names and instructions to guide routing accuracy.
🎯
Focused Prompts
Each agent keeps a short, precise instruction set focused on its specialty — no bloat, no confusion
🔀
Model Flexibility
Different agents can use different models — cheap/fast for routing, powerful for analysis, specialized for code
06
Persistence

Memory
Conversation & Context Persistence

Without memory, an agent treats every message as the first one. Memory is what makes an agent feel like an assistant rather than a glorified autocomplete.

⚡
Short-Term Memory
Holds the current conversation turn-by-turn within a single runner invocation. Automatically managed by the runner. Cleared when the session ends.
📚
Long-Term Memory
Persistent facts about the user or domain that survive across multiple sessions. Stored in an external database or vector store and retrieved on demand.
🔗
Session Memory
Workflow-scoped context shared across all agents in a single run. Every agent and tool in the same session sees the same session memory state.
Python — Memory Types
from agents.memory import SessionMemory, PersistentMemory # Short/session memory — wiped per session memory = SessionMemory() # Long-term memory — persists across sessions # (typically backed by a vector store or DB) memory = PersistentMemory(store=my_vector_db) agent = Agent( name="Assistant", memory=memory, instructions="Remember the user's preferences across our conversation." )

Memory Properties

scope
session | global
Defines visibility. Session-scoped memory is isolated per workflow run. Global memory is shared across all runs for the same user or agent, enabling genuine long-term recall.
store
MemoryStore
Backing storage provider. In-memory for development, vector databases (Pinecone, Weaviate, pgvector) for semantic retrieval, or key-value stores for exact-match lookups.
retrieval_k
int
Number of memories to retrieve per turn for long-term memory. Higher values give more context at the cost of more tokens in the prompt.
Default: 5

Shared State

Context
Shared Workflow State

Context is the shared state that travels with the entire workflow. It holds information that isn't part of the conversation itself but that every agent and tool may need — user metadata, flags, counters, and workflow variables.

Python — Context Object
# Context flows through the entire workflow context = { "user_id": 123, "premium": True, "region": "us-east-1", "language": "en", "trace_id": "req-abc-123", "step_count": 0, } result = Runner.run_sync(manager_agent, "Process order #4521", context=context)
👤
User Metadata
user_id, plan tier, region, language, timezone — anything that scopes the workflow to a specific user
🔢
Execution State
Step counters, retry counts, intermediate results, loop variables — mutable workflow bookkeeping
🏷️
Feature Flags
A/B test toggles, capability switches, environment tags (staging vs. prod) that agents and tools check at runtime
🔍
Observability
Trace IDs, span IDs, request correlation IDs — link every agent call back to the originating user request

Safety

Guardrails
Validation & Safety

Guardrails are the safety and validation layer of the SDK. They sit between the agent's output and the rest of the system, deciding what is allowed through.

Python — Custom Guardrail
from agents import guardrail, GuardrailResult @guardrail def no_pii_guardrail(output: str) -> GuardrailResult: """Block outputs containing PII like email addresses or phone numbers.""" if contains_pii(output): return GuardrailResult(approved=False, reason="PII detected") return GuardrailResult(approved=True) agent = Agent( name="Customer Agent", guardrails=[no_pii_guardrail, schema_validation_guardrail] )
approved required
bool
Pass/fail decision. True lets the output proceed downstream. False triggers the on_reject behavior (retry, refuse, or escalate).
reason optional
str
Human-readable explanation of why the guardrail rejected an output. Logged for auditing, shown to operators, or used to guide retry prompts.
on_reject optional
"retry" | "refuse" | "escalate"
Action when rejected. Retry prompts the agent to try again. Refuse returns an error to the caller. Escalate hands off to a human or supervisor agent.
Default: "refuse"
🤖
AGENT OUTPUT
Agent produces a response — text, structured output, or tool call
🛡️
GUARDRAIL CHECK
Each configured guardrail evaluates the output sequentially
✅
APPROVED
Output passes all guardrails → proceeds downstream to caller or next agent
❌
REJECTED
Guardrail blocks output → triggers retry, refuse, or escalate based on on_reject setting

Structured Output

Output Types
Typed, Reliable Responses

Output types turn agents from chatbots into APIs. Instead of free-text answers, you get typed objects you can reliably parse and pass to other systems — no fragile string parsing required.

Output types turn agents from chatbots into APIs. An agent that returns a Pydantic object is one you can wire into a backend without writing fragile parsing logic.
Python — Structured Output with Pydantic
from pydantic import BaseModel, Field from typing import Optional, List from agents import Agent class WeatherOutput(BaseModel): city: str = Field(description="City name as provided") temperature_c: int = Field(description="Temperature in Celsius") condition: str = Field(description="Weather condition (Sunny, Rainy, etc.)") humidity_pct: Optional[int] = Field(None, description="Relative humidity %") alerts: List[str] = Field(default_factory=list) agent = Agent( name="Weather Agent", output_type=WeatherOutput # SDK enforces this schema on every response ) result = Runner.run_sync(agent, "Weather in Chennai?") weather: WeatherOutput = result.final_output print(weather.temperature_c) # → 34
❌ Without output_type
"The weather in Chennai is sunny with a high of around 34°C today."
Brittle to parse, inconsistent format, not machine-readable
✅ With output_type
{ "city": "Chennai", "temperature_c": 34, "condition": "Sunny", "humidity_pct": 72, "alerts": [] }
Typed, validated, wire-ready for any downstream system

Real-Time UX

Streaming
Real-Time Token Generation

Streaming is what makes agents feel responsive. Rather than waiting for the full response, tokens arrive as they are generated — essential for chat UIs, live dashboards, and any interface where perceived latency matters.

Python — Streaming Patterns
import asyncio from agents import Runner # Pattern 1: Token-by-token stream (chat typewriter effect) async def stream_response(): async for token in Runner.stream(agent, "Write a haiku about AI"): print(token, end="", flush=True) # Pattern 2: Full workflow stream (see intermediate steps) async def stream_workflow(): async for event in Runner.run_streamed(agent, "Research quantum computing"): if event.type == "tool_call": print(f"🔧 Calling tool: {event.tool_name}") elif event.type == "token": print(event.content, end="", flush=True) elif event.type == "done": print("✅ Complete")
stream_options optional
dict
Fine-tune streaming behavior. Control buffer size, event types to surface, and whether to include tool_call events in the stream alongside token events.
RunEvent.type
"token" | "tool_call" | "handoff" | "done"
Event discriminator in run_streamed. Lets your UI differentiate between streaming tokens (show to user), tool calls (show loading spinner), handoffs (show routing indicator), and completion.

Lifecycle

Sessions
Workflow Lifecycle

A session is the lifecycle wrapper around a workflow. It ties together history, memory, state, and the continuity of a single user's interaction across multiple runner invocations.

📜
Conversation History
Full turn-by-turn log of user and agent messages within this session. Enables multi-turn coherence.
🧩
Memory State
Snapshot of memory at every step — allows replay, debugging, and rollback to earlier conversation states.
📦
Workflow Variables
Context and workflow-scoped variables persist within the session and are accessible to every agent in the run.
🔁
Cross-Invocation Continuity
Sessions persist across multiple Runner.run() calls — a user can pick up a conversation after closing and reopening the app.

Advanced

Execution Concepts
Runnables, Async, Lifecycles

Understanding Runnables and async patterns is what separates a toy agent from a production system. These concepts govern concurrency, throughput, and integration with real-world infrastructure.

The Runnable Abstraction

Runnable Type
Agent
A single AI worker with instructions, tools, and memory. The primary unit of execution in the SDK — receives input, reasons, calls tools, and produces output.
Runnable Type
Tool
An executable Python function the agent can call. Annotated with @function_tool, it becomes a first-class citizen in the SDK's execution model with schema inference and lifecycle hooks.
Runnable Type
Workflow
A choreographed sequence of agents and tools forming a complete pipeline. Can branch, loop, and conditionally hand off based on intermediate results.
Runnable Type
Chain
A linked pipeline of runnables where each step's output becomes the next step's input. Useful for sequential transformations like: fetch → parse → summarize → format.
Python — Async & Parallel Execution
import asyncio from agents import Runner # Concurrent multi-agent execution (production pattern) async def run_parallel(queries: list[str]) -> list: tasks = [Runner.run(agent, q) for q in queries] results = await asyncio.gather(*tasks) return [r.final_output for r in results] # FastAPI integration from fastapi import FastAPI app = FastAPI() @app.post("/ask") async def ask(query: str): result = await Runner.run(agent, query) return {"answer": result.final_output}
